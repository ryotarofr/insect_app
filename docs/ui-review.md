## 0. まず褒めたい設計判断

この商材は **「EC の冷たさ」と「飼育ログの温度」を両立** させる必要があり、ここはすでに上手に解けています。以下の 8 点は、レビューの前提として率直に評価します。

1. **oklch() を使った色トークン設計** (`styles/tokens.css`): 知覚均等な色空間で `--accent-forest/amber/rose/indigo` を系統的に定義していて、カブクワ商材の「落ち着いた森のトーン」が数値で担保されている。琥珀・森緑・藍の配色は分類学的な「標本ラベル感」と EC の信頼感を両立できている。
2. **専門記法の尊重**: 学名の `<em>` italic 組み (`ProductDetailContent.tsx:44`)、「CBF2」「WF1」などの累代記法、`142mm (頭角含)` などの採寸注記。ブリーダー層が一目で「わかってる UI」と判断できる。`mono` クラスを ID・寸法・日付に徹底しているのも良い。
3. **羽化予測というプラットフォーム固有の情報設計**: Hero / サイドバー Badge / 商品カードに一貫して羽化日を露出している。これは汎用 EC には無い、この商材の中核価値を UI で言語化できている好例。
4. **Shell の nav-group 分類**: `EC / 飼育 / 取引 / 運営` の 4 グループ化 (`Shell.tsx:17`) は、ユーザータイプ (買い専 / 飼育家 / ブリーダー / ショップ) の遷移メンタルモデルと一致している。
5. **セリフ×モノスペースのタイポシステム**: 見出しのセリフ (個体名・学名) とモノ (ID・日付・サイズ) の役割分担が明確で、情報密度が高い画面 (個体カルテ等) でも読み分けできる。
6. **Hero の 3 Promises の語り口**: `買う / 育てる / 継ぐ` のコピーは、単なる機能紹介ではなく「この商材がなぜ存在するか」を述べている。EC にしては珍しく説得力がある (`pages/products/Hero.tsx`)。
7. **ドメインデータ API の抽象化**: `api.ts` が `getProduct` / `listProducts` / `getCurrentUser` 等で分離され、モック→本番差し替えが容易。今のフェーズでは妥当な設計。
8. **キーボードショートカット 1-9 での画面切替** (`App.tsx:113-123`): PC での濃いユーザー (ショップ運営者) への配慮として効く。これは iOS 版と差別化できる Web 固有の UX。

## 1. P0 / P1 重大指摘

### 1-1. viewport が `width=1280` 固定 — モバイル設計が完全に死んでいる [P0 / S]

- **該当箇所**: `client_solid/index.html:5`
  ```html
  <meta name="viewport" content="width=1280" />
  ```
- **現状**: すべてのモバイル端末で PC レイアウトが 1280px 幅で縮小表示される。文字は小さく、タップターゲットは 10px 台になり、`app-layout.css` にある 640px・380px 向けの bottom-tab 実装 (lines 975-1093) が**一度も発火しない**。
- **問題**:
  - 片手操作・屋外 (採集・飼育室) 前提の本商材で、モバイル UX が実質「拡大/縮小ピンチ操作を強いる 2000 年代サイト」になっている。
  - PWA を謳うのに、iOS ホーム画面追加後もスマホで 1280px 表示される。
  - `app-layout.css` の 400行 近い mobile CSS が死資産になっている。
  - 影響ユーザー: **全モバイルユーザー (推定 60% 以上)**。
- **提案**:
  1. **(推奨) 標準の viewport に戻す**:
     ```html
     <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
     ```
     `viewport-fit=cover` は iOS の safe-area (notch/home-indicator) を使うために必要。
  2. それでも PC デザイン崩れが怖い場合は、デスクトップ最小幅を CSS 側で `min-width: 320px` にしつつ、`html { min-width: min(100vw, 1280px); }` のような段階フォールバックに変える。ただし**どんな場合でも `width=1280` 固定は不可**。
- **優先度**: **P0** — これを直さないと他のモバイル指摘が全部机上の空論になる。
- **工数感**: S (30分 + モバイル実機確認)。

### 1-2. PWA が「名乗っているだけ」 — manifest / service worker が未実装 [P0 / M]

- **該当箇所**: リポジトリ全体 (`manifest.json`, `service-worker.*`, `vite-plugin-pwa` 等が見当たらない — 未確認だが grep ヒット 0)
- **現状**: プラットフォーム名が "PWA" でも、ホーム画面アイコン・スプラッシュ・オフライン対応・Background Sync が無い。`QuickLogSheet` でオフライン時に記録した飼育ログの保全ができない。
- **問題**:
  - 飼育室・屋外 (電波不安定) での記録が本商材の核。オフライン投函→復帰時同期できない。
  - iOS アプリと共通の「片手でログ」体験が、Web では提供できない。
  - `Hero.tsx` の「累計カルテ 12,480 件」などの訴求と矛盾する。
- **提案**:
  1. **最小構成**: `vite-plugin-pwa` を入れて manifest.json (icon 192/512, theme_color `var(--bg)` 相当, display: "standalone") と Workbox の precache を導入。これで「ホーム画面追加」「オフラインで画面だけは開く」が実現する (工数 M)。
  2. **ドメイン特化**: 上記に加えて、飼育ログだけは IndexedDB キュー + Background Sync で非同期送信する専用 Service Worker を書く (工数 L)。最優先は `QuickLogSheet` の送信パス。
- **優先度**: **P0** — 商材的に致命的。ただし 1-1 を先に直すこと。
- **工数感**: M (最小構成) 〜 L (Background Sync 込み)。

### 1-3. フォームの `font-size: 13px` — iOS Safari で入力時に毎回ズームされる [P0 / S]

- **該当箇所**: `styles/tokens.css:163-178` (`input, select, textarea { font-size: 13px }` 系)
- **現状**: iOS Safari は input の `font-size < 16px` だとフォーカス時に自動ズームし、戻らない。検索・チェックアウトのすべての入力で起きる。
- **問題**:
  - チェックアウト (`Cart.tsx`) で離脱の主要因になる。
  - `QuickLogSheet` の温度入力 `28.4` のような即時入力ユースでは特に操作感が悪化。
  - 1-1 を直した後に一気に顕在化する。
- **提案**:
  1. **最小修正**: input 系の `font-size` を `16px` に上げる。見た目が崩れる場合は `transform: scale(0.875)` で視覚的に縮め、`transform-origin: left center` で配置調整する。
     ```css
     input, select, textarea { font-size: 16px; }
     ```
  2. **メディアクエリで分岐**: `@supports (-webkit-touch-callout: none)` で iOS のみ 16px、それ以外は 13px。
- **優先度**: **P0**。
- **工数感**: S (1-2h)。

### 1-4. URL ルーティング不在 — 共有・戻る・SEO が全て壊れている [P1 / M]

- **該当箇所**: `App.tsx:93` `const [route, _setRoute] = createSignal<RouteKey>(initialRoute);` と localStorage `kochu:route` 保存
- **現状**: ルートは URL に出ず、`localStorage` にのみ保存される。ブラウザの戻る/進むは効かない。商品 URL を共有しても「前回開いた画面」が復元される。
- **問題**:
  - EC において**商品 URL を共有できないのは致命的**。SNS/LINE での拡散経路が死ぬ。
  - iOS ネイティブアプリから Web に誘導 (Universal Link) する際の deep link ターゲットが無い。
  - `Bloodline.tsx` の個体選択も URL に残らず、SNS で「この血統見て」ができない。
  - SEO クローラは SPA 内部を辿れず、インデックスが Hero 画面のみになる。
- **提案**:
  1. **(推奨) `@solidjs/router` を導入** し、`/products`, `/products/:id`, `/bloodline/:specimenId`, `/log/:specimenId` 等の URL 体系にする。商品は SSG/prerender で静的化できると SEO も改善。
  2. **最小代替**: 既存の signal ルーティングを維持しつつ、`history.replaceState` で `?r=products&p=p-hh-m-142` のようなクエリ同期だけ足す。数時間で実装可能だが、後で 1 に移行する前提で。
- **優先度**: **P1** (EC としては P0 相当だが、フェーズを考慮して P1)。
- **工数感**: M — ルータ導入自体は 1 日、各ページのルート整理と product/specimen 詳細の URL 化で +1 日。

### 1-5. `Cart.tsx` の配送先が氏名・電話ハードコーディング [P0 / S]

- **該当箇所**: `Cart.tsx:162-189`
  ```tsx
  <input id="ship-name" class="input" value="山田 徹" />
  <input id="ship-tel"  class="input mono" value="080-0000-0000" />
  ```
- **現状**: 配送先欄に固有名・ダミー電話番号がハードコードされていて、`onInput` も無い。実質 readOnly。
- **問題**:
  - 本番チェックアウトで決して許容できない。入れ替わり配送のリスク。
  - デモ時も「山田徹」が他ユーザーの画面に出続けるのは信用を損なう。
  - 生体の配送は**温度制御便で一発勝負** (Hero にも明記) なのに、住所編集 UX が死んでいる。
- **提案**:
  1. 最低限、`value` を `getCurrentUser()` から引く **signal 連動の `Input`** に変え、`onInput` で `store/checkout` へ同期する。未ログインなら空欄にする。
  2. 配送先は複数保存 (実家・事務所・ブリーダー出張先) の選択にし、`select` + 新規追加ボタンにする。生体購入時は「配送先が冷暖房可能か」を確認するチェックもこのタイミングで聞くと、ドメインに効く。
- **優先度**: **P0**。
- **工数感**: S (半日)。ただし store 層まで足すと M。

### 1-6. `Bloodline.tsx` が渡された specimen を無視して `#DHH-0271` 固定 [P1 / S]

- **該当箇所**: `pages/Bloodline.tsx:771` 付近
  ```ts
  const [selected, setSelected] = createSignal("#DHH-0271");
  ```
- **現状**: 他画面 (マイページ・個体カルテ) から「この個体の系図を見る」で飛んでも、常に同じサンプル個体が表示される。
- **問題**:
  - 血統系図はこの商材の花形機能。そこが**常に同じ個体**というのは、ユーザーが一番最初に気づく致命バグ。

- **優先度**: **P1**。
- **工数感**: S (1-2h)。

### 1-7. 「死着 24h 自動補償」の導線が UI shell しか無い [P1 / M]

- **該当箇所**: `Hero.tsx:209-220` (`onLink={() => props.setRoute("market")}`)
  ```tsx
  title="死着 24h 自動補償"
  ...
  link="→ 補償フローの詳細"
  onLink={() => props.setRoute("market")}
  ```
- **現状**: Hero の「3つの約束」で **3 番目に重要な差別化ポイント**である補償フローを押すと、なぜか C2C マーケット画面に飛ぶ。
- **問題**:
  - ランディング最後の信頼要素がクリックされて「マーケット画面」に着地するのは、ユーザーが「え？」となる最悪のデッドエンド。

- **優先度**: **P1** (信頼の根幹)。
- **工数感**: M (専用ページ) 〜 S (暫定リンク化)。

### 1-8. ナビ項目が `<div role="button">` — 本物の button/a を使っていない [P1 / S]

- **該当箇所**: `Shell.tsx:79-91`
  ```tsx
  <div class={"nav-item" + ...} role="button" tabindex="0"
       onClick={...} onKeyDown={...}>
  ```
- **現状**: サイドバーのナビを `div` + `role="button"` + 手書き Enter/Space 判定で実装している。
- **問題**:
  - 本来**ページ遷移**なので `<a href>` が正しい。現状は (a) スクリーンリーダーが「リンク」と読まない (b) 右クリック「新しいタブで開く」が効かない (c) キーボード操作が手書きに依存。
  - Space キーで button は発火する一方、Enter も想定され自作ロジックが必要になり、将来の修正でバグりやすい。
  - アクセシビリティと URL ルーティング (1-4) 両方の一石二鳥になる改修ポイント。
- **提案**:
  1. `@solidjs/router` 導入と合わせて `<A href="/products" classList={{ active: isActive() }}>` に置き換える。`aria-current` は router が付けてくれる。
  2. 一旦 router 無しでやるなら `<a href={`#${n.key}`} onClick={e => { e.preventDefault(); setRoute(n.key); }}>` で最低限の「右クリックで URL が見える」「Cmd+Click で新タブ」を獲得する。
- **優先度**: **P1**。
- **工数感**: S (1-2h)。

### 1-9. トップバーの検索が `disabled` のまま常時露出 [P1 / S]

- **該当箇所**: `Shell.tsx:121-147` (`<input ... placeholder="個体ID・種名・商品を検索（準備中）" disabled>`)
- **現状**: 全画面のヘッダ中央に、グレーアウトした検索窓と `⌘K` が見える。

### 1-11. モーダル系の focus trap が未実装 [P1 / S]

- **該当箇所**: `QuickLogSheet.tsx` (role="dialog", aria-modal="true") / Market 出品モーダル等
- **現状**: `role="dialog"` と `aria-modal="true"` は付与されているが、開閉時の **focus 移動・Tab の閉じ込め・Esc クローズ・背景 inert** が実装されていない。
- **問題**:
  - キーボードユーザーが、Tab で背景の要素にフォーカスが抜けてしまう。
  - スクリーンリーダーで背景が読める。
  - `QuickLogSheet` はこの商材の核ユースケース (飼育室で手早く記録) なので、ここが特にクリティカル。
- **提案**:
  1. 小さな `createFocusTrap(ref)` ユーティリティを書く (openedRef にフォーカス移動 → Tab のループ → Esc で close)。Solid の `onMount`/`onCleanup` で着脱。
  2. `inert` 属性対応 (Safari 15.5+/Chrome 102+) なら、背景コンテナに `inert` を付けるだけで相当楽になる。未対応ブラウザフォールバックとして 1 を併用。
- **優先度**: **P1** (A11y + 片手操作時の誤タップ防止)。
- **工数感**: S (1 日)。

### 1-12. モバイル bottom-tab が実装されているが、viewport 修正で初めて活きる [P1 / M]

- **該当箇所**: `app-layout.css:975-1093`、`Shell.tsx` は sidebar 前提
- **現状**: CSS 上の 640px 以下で sidebar を bottom-tab 化するスタイルは書かれているが、`Shell.tsx` のマークアップが sidebar のままなので、表示順序・ロールが合わず iOS で違和感が出る恐れ。
- **問題**:
  - 1-1 viewport 修正後、モバイルでようやくこの CSS が動くが、nav-group (EC/飼育/取引/運営) が bottom-tab にそのまま縦展開されるのは想定と違う可能性が高い。

- **工数感**: M (1-2 日)。viewport 修正後すぐ着手。

### 1-13. `MyPage.tsx` の KPI 数値がハードコード [P1 / S]

- **該当箇所**: `MyPage.tsx:25-30` (血統ライン 4, 今月ログ 28 など)
- **現状**: ダッシュボードの指標が固定値。
- **問題**:
  - 新規ユーザーが開いても「血統ライン 4」と表示される。**デモ感が強く信頼を損なう**。
  - 本商材はマイページ = カルテ集約の「帰る場所」で、ここが嘘数値だと全体の信頼が落ちる。
- **提案**:
  1. `api.ts` に `getUserMetrics()` を生やし、`listSpecimens().length` 等の派生値を createMemo で計算。データが無いユーザーには「まだカルテがありません・生体を探す」を出す (空状態と連動)。

## 4. 全体サマリー

### 強み
- **ドメイン設計の解像度が高い**: 学名・累代・羽化予測・血統 F 値までが UI に載っており、汎用 EC テンプレを飛び越えて「この業界のための道具」を目指しているのが伝わる。
- **デザイントークンとタイポ**: oklch 色空間 + セリフ/モノ併用は、標本ラベルの質感を現代の Web UI に翻訳できている稀有な試み。
- **ドキュメント構成・命名**: ページ/コンポーネント分割が `pages/<domain>/<Piece>.tsx` で素直。スケールする前提の構造になっている。
- **Hero コピー**: 「買う・育てる・継ぐ」の 3 本柱は、長期リテンション型商材の価値訴求として正しい。

### 構造的な弱点
1. **モバイル前提が形骸化**: viewport=1280 固定 (1-1)、PWA 未実装 (1-2)、font-size 13px (1-3) で、モバイル UX の基礎が揃っていない。片手操作・屋外という本商材のコアユースケースが Web 側で成立していない。
2. **URL と状態の分離**: SPA の中で `localStorage` のみに保存される route/selectedSpecimen のせいで、共有・戻る・SEO・Universal Link がすべて崩壊している (1-4, 1-6)。
3. **ハードコードされたダミーデータが本番 UI に露出**: 山田徹 (1-5)、MyPage の数値 (1-13)、QuickLog 28.4 (1-14)、Hero 99.2% (2-3) など、信頼の根幹を揺さぶる。
4. **"Promise" と実装のギャップ**: Hero で謳う「カルテ生成」「血統」「補償」のうち、補償は動線不在 (1-7/1-16)、血統は固定 (1-6) という**一番見られる場所で約束が破られている**。
5. **ネガティブイベントの UX 欠落**: 死亡・脱走・羽化不全など、実際の飼育で最も重いイベントが設計されていない (1-15)。これはドメイン UX の根幹。

### 今スプリントでやるべき Top 5 (推奨順)

| # | 項目 | P | 工数 | 効果 |
|---|------|---|------|------|
| 1 | viewport = device-width / iOS font-size 16px | P0 | S | モバイル UX の基礎が一気に立ち上がる。他の指摘の前提条件。|
| 2 | Cart 配送先のハードコード除去 + onInput 接続 | P0 | S | 決済が実質機能していない致命バグを解消。|
| 3 | `@solidjs/router` 導入 + nav を `<a href>` 化 | P1 | M | URL 共有・戻る・Universal Link・a11y が一気に進む。|
| 4 | Bloodline が specimenId を受ける + Hero の補償リンク修正 | P1 | S | 「約束の破れ」を止める。Hero→詳細→系図の動線復活。|

### iOS ネイティブとの一貫性について (Web 側の割り切り)
- Web は **発見・共有・詳細閲覧・SEO** を主、iOS は **ログ入力・通知・オフライン** を主に役割分担する前提で、このレビューは Web で `@solidjs/router` を必須として扱った (Universal Link の着地点を揃えるため)。
- 一方、`QuickLogSheet` や FAB のような入力系は Web 上で無理に iOS と完全一致させる必要はない。Web では **"とりあえず開ける" 記録 UI** 程度にし、本命は iOS に寄せる設計が合理的。

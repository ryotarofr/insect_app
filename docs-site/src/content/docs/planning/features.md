昆虫EC × 飼育管理プラットフォームの全機能を、2年間のソロ開発ロードマップに沿って整理したドキュメント。MVP（4ヶ月）から Phase 8（24ヶ月）までを対象とする。

## 全体俯瞰

| Phase | 期間 | テーマ | 主要機能 |
|-------|------|--------|---------|
| MVP | M1-M4 | 土台 + コア体験 | 基盤・EC・自動カルテ・飼育ログ・羽化予測 |
| Phase 1 | M5-M6 | 検索とAPI整備 | 全文検索、OpenAPI完全化 |
| Phase 2 | M6-M9 | モバイル展開 | iOSアプリ、APNs プッシュ通知 |
| Phase 3 | M9-M11 | 生体保護の強化 | 動画アップロード、死着補償自動化 |
| Phase 4 | M11-M14 | ブリーダー基盤 | 血統管理UI、ブリーダー認証・スコア |
| Phase 5 | M14-M17 | C2C 経済圏 | C2C マーケットプレイス |
| Phase 6 | M17-M19 | 自動化とAI第一歩 | 消耗品自動補充、AI種同定 |
| Phase 7 | M19-M21 | コミュニティ化 | フィード、マルチショップ対応 |
| Phase 8 | M21-M24 | スマート化 | IoT連携、AI相談・高精度予測 |

## Phase 0: MVP (M1-M4)

### 基盤レイヤー（foundation）

#### F1. マルチテナント対応スキーマ

- 全エンティティに `shop_id` を持たせ、テナント単位で分離可能な設計
- MVP時点ではアンカー1店舗固定だが、Phase 7 で活用
- PostgreSQL の Row-Level Security 導入も検討
- 完了条件：商品・注文・個体が shop_id で分離されている

#### F2. イベントソーシング基盤

- 個体状態変化、血統登録、取引を不変のイベントログとして保存
- `events` テーブル：`id, aggregate_id, aggregate_type, event_type, payload (JSONB), created_at, actor_id`
- C2Cの真贋保証、監査、血統改ざん防止の基盤
- 現在状態はイベントから導出、または派生ビューで保持
- 完了条件：個体・血統・取引の全変更がイベント経由で記録される

#### F3. OpenAPI コード生成パイプライン

- Axum ハンドラに `utoipa` 属性を付けて OpenAPI スキーマ生成
- CI で `openapi-typescript` により Solid.js 用型を自動生成
- iOS 用の `swift-openapi-generator` 呼び出しも仕込んでおく（iOS アプリは Phase 2）
- 完了条件：API変更時に型定義が自動的に同期される

#### F4. R2 ダイレクトアップロード基盤

- クライアントから Cloudflare R2 へ直接アップロードする署名URL発行
- サーバーはメタデータ管理と完了通知受信のみ
- 画像ファイル形式制限、サイズ上限、ウイルススキャン（将来）
- 完了条件：商品画像・個体写真・飼育ログ写真が統一フローで扱える

#### F5. バックグラウンドジョブ（apalis）

- PostgreSQL バックエンドのジョブキュー
- 用途：羽化予測通知、メール送信、将来の自動補充・AI推論
- リトライポリシー、冪等性確保、デッドレター処理
- 完了条件：予測通知が定時実行されメールが届く

#### F6. 構造化ログと Sentry

- `tracing` で構造化ログ出力、JSON形式で集約
- Sentry でエラー追跡、パフォーマンス計測
- ログにリクエストID、ユーザーID、セッションIDを付与
- 完了条件：本番障害が Sentry で検知・通知される

#### F7. CI/CD

- GitHub Actions or 同等：テスト・Lint・ビルド・デプロイ自動化
- マイグレーション自動実行（dev/staging/prod）
- プレビュー環境（PRごと）
- 完了条件：main へのマージで本番デプロイが完了する

#### F8. 認証基盤（JWT + OAuth ready）

- メール+パスワード認証（argon2 ハッシュ）
- JWT アクセストークン + リフレッシュトークン
- OAuth 設定済み（Google、Apple Sign In は実装のみで呼び出しは iOS リリース時）
- ロール：user / breeder / shop_admin / platform_admin
- 完了条件：ログイン・セッション維持・ロール別アクセス制御が動作する

### EC コア

#### E1. 商品カタログ

- 商品種別：生体（個体固有）、用品（在庫型）
- 生体は1点1個体として登録（サイズ、血統、羽化日付）
- 用品は在庫数管理（ゼリー、マット、菌糸ビン、ケース等）
- カテゴリ、タグ、画像複数枚
- 完了条件：ショップが商品登録でき、購入者が一覧・詳細を閲覧できる

#### E2. カート・チェックアウト

- カート管理（ゲストカートはMVPでは無し、ログイン必須）
- 配送先入力・保存
- 配送料計算（地域・重量・温度制御の簡易版）
- Stripe Checkout 連携（決済情報は保持しない）
- 完了条件：カートに商品追加→決済完了→注文確定までが通る

#### E3. 注文管理

- 注文ステータス：pending / paid / preparing / shipped / delivered / cancelled
- 注文履歴閲覧
- ショップ側：注文一覧、ステータス更新、トラッキング番号入力
- キャンセル・返金は MVP では手動対応（Stripe管理画面から）
- 完了条件：購入者・ショップ双方が注文状況を把握できる

#### E4. ショップ管理画面

- 商品CRUD（画像アップロード含む）
- 個体登録（生体）：種・サイズ・羽化日・血統・価格
- 注文一覧・ステータス更新
- 配送情報入力
- 完了条件：ショップが自走で日次運用できる

### 飼育管理コア

#### K1. 個体自動カルテ生成

- 生体購入時に `specimens` テーブルへ自動レコード生成
- 商品から引き継ぐ情報：種、サイズ、羽化日、血統、購入日、購入元ショップ
- 購入者の所有個体として紐付け
- 完了条件：生体購入後、購入者のマイページにカルテが出現する

#### K2. カルテ詳細画面

- 基本情報表示（自動生成分）
- 編集可能項目：名前、メモ、ステータス（生存/死亡/譲渡）
- 写真複数枚アップロード
- 飼育ログへのリンク
- 完了条件：個体ごとの記録を一覧閲覧できる

#### K3. 血統データ構造（データのみ）

- `bloodlines` テーブル：親個体への参照（父親・母親）、累代数
- MVPでは UI は簡素（親個体IDを手動入力するフォーム）
- 系図可視化は Phase 4 で追加
- 完了条件：親子関係をデータとして記録できる

#### K4. 飼育ログ

- エントリ型：体重計測、脱皮、マット交換、エサ補給、異常観察、その他
- 各エントリに日時、値（体重など）、写真、メモ
- タイムライン表示（個体ごと）
- 完了条件：個体の履歴が時系列で確認できる

### 通知

#### N1. ルールベース羽化予測

- `species_stats` テーブル：種別の平均羽化日数（幼虫期・蛹期）
- 個体の現在ステージから羽化予測日を算出
- カルテに予測日表示
- apalis ジョブで日次チェック、「羽化7日前」でメール通知
- 完了条件：予測日がカルテに表示され、近づくとメールが届く

#### N2. メール通知

- Resend or SendGrid で送信
- トリガー：注文確認、発送通知、羽化予測リマインド、パスワードリセット
- テンプレート管理（shoppedとsystemで分離）
- 完了条件：各トリガーで適切なテンプレートメールが届く

## Phase 1: M5-M6（検索とAPI整備）

### S1. 全文検索（Meilisearch）

- Meilisearch をセルフホストまたは Meilisearch Cloud
- インデックス対象：商品、個体カルテ（所有者のみ）、ショップ、種名

- 全エンドポイントに utoipa 属性付与
- 型の一貫性チェック（CI）
- iOS 用 Swift クライアント生成スクリプトを走らせられる状態に
- エラーレスポンスのスキーマ統一
- 完了条件：iOSアプリ開発の準備が完了している

## Phase 3: M9-M11（生体保護の強化）

### V1. 動画アップロード

- Cloudflare Stream or Mux 連携
- クライアント（Web/iOS）から直接アップロード、サーバーはメタデータのみ
- 自動トランスコード（複数解像度）、ストリーミング再生
- 用途：開封動画、個体紹介、羽化記録
- 動画長制限（MVP は 60秒まで）
- 完了条件：アップロード→視聴までが5分以内に完了

### D1. 死着補償自動化

- 配送到着時の開封動画アップロードを購入者に促す
- 動画と注文データを紐付け
- 検証フロー：
  - 自動チェック（商品情報、動画尺、タイムスタンプ）
  - 死亡が認められれば自動で Stripe 返金処理
  - グレーゾーンは管理者承認フローへ
- 完了条件：明白な死着は24時間以内に自動返金される

## Phase 4: M11-M14（ブリーダー基盤）

### B1. 血統管理UI（系図可視化）

- 既存の `bloodlines` データを系図として可視化
- インタラクティブなツリー表示（D3.js or 類似）
- 交配記録入力フロー：父・母個体を選択→子個体自動生成
- 累代数の自動計算
- 血統書PDF出力
- 改ざん防止：変更履歴をイベントログで保持
- 完了条件：ブリーダーが自分の繁殖系図を直感的に管理できる

### B2. ブリーダー認証・スコアリング

- 認証申請フロー：実績入力、書類アップロード（身分証、飼育歴）
- スコア算出要素：
  - 累代数の深さ
  - 羽化実績数
  - 飼育継続月数
  - ユーザーレビュー平均
  - 死亡率（低いほど加点）
- プロフィール画面にバッジ表示
- 管理者レビュー画面
- 完了条件：上位50名のブリーダーが認証済み状態で運用できる

## Phase 5: M14-M17（C2C 経済圏）

### C1. C2Cマーケットプレイス

- 出品機能：
  - 自分の個体を選んで出品（カルテ情報が自動で出品説明に反映）
  - 価格、即決/オークション、期限設定
  - 写真・動画必須
- 購入・入札機能：
  - ウォッチリスト
  - 入札履歴
  - 即決購入
- エスクロー（代金預かり）：
  - Stripe Connect で売り手別アカウント
  - 購入者支払い→プラットフォーム預かり→到着確認後に売り手へ支払い
  - 死着時は自動返金
- メッセージング：
  - 出品ごとのスレッド
  - 画像添付可、テキスト中心
  - 通報機能
- レビュー：
  - 取引完了後に相互レビュー
  - ブリーダースコアに反映
- 血統真贋保証：
  - 認証済みブリーダーの累代個体にバッジ
  - 血統履歴がイベントログで検証可能
- 完了条件：手数料収入が発生、月次取引100件以上

## Phase 6: M17-M19（自動化とAI第一歩）

### R1. 消耗品自動補充提案

- ユーザーの飼育個体数 × 種別 × 期間から消費予測
- 消耗品別（ゼリー、マット、菌糸ビン）の残量推定
- 残量低下時にプッシュ通知＋ワンタップ発注フロー
- ユーザー設定：自動発注、提案のみ、停止
- 完了条件：消耗品リピート率が Phase 5 比で30%向上

### AI1. AI種同定

- カメラで個体撮影→種名・性別・推定サイズを判定
- Google Cloud Vision / AWS Rekognition / 自社モデルのいずれか
- 対象種：主要カブクワ30種から開始
- 信頼度スコア表示
- 不確実時は上位3候補提示
- 完了条件：主要30種で精度80%以上

## Phase 8: M21-M24（スマート化）

### IoT1. IoT連携

- 温湿度センサーとの連携（BLE経由 or WiFi）
- 対応デバイス：SwitchBot 温湿度計（API連携）、その他主要製品
- 時系列データストレージ（TimescaleDB 拡張）
- 異常値アラート（閾値カスタマイズ）
- 異常時の推薦商品表示（保温器具、エアコン等）
- 完了条件：1デバイス以上から5分以内にデータ取得、異常時に通知

### AI2. AI相談

- チャットUI（テキスト + 画像）
- OpenAI or Claude API を利用
- RAG：飼育Wiki、過去の相談データを参照
- 用途：飼育トラブル相談、病気診断、初心者Q&A
- エスカレーション：深刻なケースは認証ブリーダー/獣医へ案内
- 完了条件：月間相談件数500件以上、満足度80%以上

### AI3. AI高精度羽化予測

- 実績データ（過去2年分）を使ってMLモデル学習
- 入力：種、サイズ、温度、飼育条件、ログ履歴
- 出力：羽化日の予測幅（例：2026-07-15 ±3日）
- ルールベース予測との比較ダッシュボード
- 完了条件：ルールベース比で誤差50%削減

## 機能優先度マトリクス

「ユーザー価値の高さ × 実装コストの低さ」で見たときの優先度。MVPは左上、その他は価値か効率で選定。

| 高価値・低コスト | 高価値・高コスト |
|-----------------|-----------------|
| 自動カルテ、羽化予測、メール通知 | C2Cマーケット、iOSアプリ、血統系図UI |

| 低価値・低コスト | 低価値・高コスト |
|-----------------|-----------------|
| コミュニティ投稿（単機能） | IoT連携、AI高精度予測 |

## 開発上の不変方針

2年を走り切るための設計原則。

1. **データは不変、ビューは可変**：イベントソーシングを守る
2. **外部サービスは疎結合**：Stripe/R2/Meilisearch/APNs は Adapter 層で隔離
3. **週次デプロイ**：機能未完成でもフィーチャーフラグで隠してデプロイ
4. **毎月1週のリファクタ週**：技術的負債返済を予算化
5. **E2Eテストは主要フローのみ**：全ユニットテスト主義にしない、コスパで選ぶ
6. **機能追加は常にマイグレーション付き**：DB変更を後回しにしない
